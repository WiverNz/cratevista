# CrateVista PRD Index

## Product summary

**CrateVista — Interactive Rust Architecture & Documentation Explorer.** A standalone open-source Cargo subcommand (`cargo cratevista`) that turns any Rust workspace into an interactive, browser-based architecture and documentation map. It ingests `cargo metadata` and rustdoc JSON, merges them (plus optional manual flows/overrides) into a stable, versioned explorer document, and serves a prebuilt React Flow SPA — locally, source-private, with no end-user Node.js requirement.

These PRDs convert `ISSUES/issue_*.md` into implementation-ready plans. **PRDs 01–09 are Implemented / Verified**; **PRD 10 is Approved — safe to implement**. PRD 01 delivered the workspace + `cargo-cratevista` CLI + `cratevista-core` scaffolding (edition 2024 / latest-stable Rust 1.97, ADR-0010; `MIT OR Apache-2.0`; `target/cratevista/` output layout + `build --output`). PRD 02 delivered `cratevista-schema` (`ExplorerDocument` + separate `generation.json`/`diagnostics.json` artifacts, `DocumentDiagnostic`, schemars 1.x + one canonical JSON serializer, blake3 semantic ID discriminators, open kinds, no schema→core dependency). PRD 03 delivered `cratevista-metadata` (Cargo metadata ingestion → schema entities/relations). PRD 04 delivered `cratevista-rustdoc` (rustdoc JSON ingestion → schema entities/relations) against the empirically verified compatibility tuple recorded in ADR-0004 (nightly `nightly-2026-07-01` → `format_version 60` → `rustdoc-types 0.60.0` → adapter `1`). PRD 05 delivered the pure `cratevista-graph` builder + `build_rustdoc_plan` and the `cratevista-core` `run_generate` orchestration/prepare-then-commit artifact writing + `generate` CLI wiring that produce `document.json`/`generation.json`/`diagnostics.json`. Implement one PRD at a time after approval, keeping each PRD's acceptance checklist synchronized with the code.

- **`cratevista-core` is an orchestration/use-case crate only** (decision). It composes the analyzer/graph/server crates into the `generate`/`serve`/`build` use cases and hosts application-runtime scaffolding (top-level error, terminal diagnostic rendering, logging, exit codes, process/OS path resolution, use-case entry points). It must **not** become a dumping ground for shared domain models — those live in `cratevista-schema`. Domain path validation (validated repo-relative `SourceLocation`, traversal safety) lives in `cratevista-schema::source` and is reused by server/config.
- **Created in issue 01:** `cratevista-core`, `cargo-cratevista`, and compiling placeholders for `cratevista-{schema,metadata,rustdoc,graph,server}`. **`cratevista-config` and `cratevista-watch` are NOT created in issue 01** — they are owned by issues 08 and 09 respectively.
- `cratevista-{core,config,watch}` are **additions** beyond CLAUDE.md's crate list (which permits boundary adjustment via an approved PRD). See "Decisions requiring approval."

## Major cross-cutting decisions

Applied consistently across PRDs. Items marked **(approval)** are still open; items marked **(approved)** were confirmed by the user with PRD 01.

1. **Extra crates** `cratevista-core` (orchestration-only), `cratevista-config` (issue 08), `cratevista-watch` (issue 09), beyond CLAUDE.md's list. Config/watch are created by their own issues, not issue 01. **(approved)**
2. **Schema compatibility & determinism** (issue 02): root type `ExplorerDocument`; no React Flow types, no rustdoc numeric ids; stable ids from names/canonical paths with **blake3** (128-bit, documented-format) semantic discriminators for impl signatures and external-source disambiguation. **Entity/relation kinds are open** (string-backed with known constants); **unknown kinds are preserved and rendered by a generic frontend fallback**, so **adding a kind is additive (MINOR)**, and there is **no `--strict` mode**. **Breaking changes are structural/semantic incompatibilities** (removing/renaming/retyping a field, changing a field's/kind's meaning, changing the id scheme or a blake3 input format). **Three separate top-level artifacts**: deterministic `document.json` (`ExplorerDocument`, no timestamps), `generation.json` (`GenerationReport`), `diagnostics.json` (`DiagnosticsReport` of `DocumentDiagnostic`) — diagnostics are **not** embedded in the document. One **canonical JSON serializer** (recursive key ordering, preserved array order, one trailing newline) serializes every artifact and the schema; **schemars 1.x** JSON Schema is generated and checked in, guarded by a drift test. `cratevista-schema` does **not** depend on `cratevista-core`. **(approved)**
3. **Stable id scheme**: `kind:canonical_path` forms (e.g. `package:{name}`, `item:{kind}:{crate}::{path}`); versions/spans excluded from ids so they survive bumps.
4. **CLI surface & exit codes** (issue 01): global `--manifest-path`, `-v/--verbose`, `-q/--quiet`, `--color`, `--format human|json`; `generate` gains `--keep-going` (issue 05); `build` gains `--output` (issue 10). `--no-cache` is **not** in the CLI today and is deferred to **`ISSUES/issue_12_persistent_cache.md`** (persistent cache), **not** to PRD 09 — watch mode ships without a cache, so PRD 09 adds no `--no-cache`.
